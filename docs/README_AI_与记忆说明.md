# Reado：AI 能力说明（无 Agent 架构 & 「AI 导师为什么像记得上下文」）

面向产品 / 老板的说明文档：**当前产品没有使用 AG（Agent / 智能体编排）那一套**，但 **AI 导师** 和 **AI 拆解** 仍然可以效果很好。下面分开说原理，并专门回答：**为什么问 AI 导师时，感觉大模型「记得上一句在问什么」？**

---

## 1. 先说结论：谁在起作用？

| 能力 | 本质 | 「记忆」从哪来 |
|------|------|----------------|
| **AI 导师（卡片内对话）** | 每次调用都是 **一次独立的 API 请求**，把 **整段对话 + 卡片正文** 写进 **同一个超长提示词** | **你们 App + Firestore**：不是 Google 替你「记着会话」 |
| **AI 拆解 / 生成卡片** | **单次（或流水线）大提示词 + 结构化要求**，后台任务跑完写库 | **无多轮对话**；效果来自 **提示词工程 + 输入材料** |

**没有使用：** 多步 Agent（自动规划 → 调工具 → 再规划）、长期 Agent 记忆库、向量库 RAG 管道（若以后加再另说）等。

---

## 2. AI 导师：为什么像「记得上一句」？

### 2.1 真相：模型端 **没有** 跨请求的长期会话

每次点发送，对 Gemini 来说都是 **新的一包输入**。服务端 **不会** 默认帮你保存「上一轮聊天」——除非你在 **下一次请求里自己带上去**。

### 2.2 你们实际做法（代码里就是这样）

1. **前端维护消息列表**  
   用户和 AI 导师的每一条消息都存在内存里的 `_messages`。

2. **每次发新问题前，把整个列表塞进提示词**  
   在 `ContentGeneratorService.chatWithContentStream` 里会把 `history` 拼成一段文字，例如：

   - `用户: …`
   - `AI Mentor: …`
   - `用户: …`（最新一条）

   再和 **当前卡片的正文（背景内容）** 一起放进 **同一个 prompt**，让模型 **在这一轮里** 同时看到：  
   「卡片讲了什么」+「之前聊过什么」+「现在用户最新问的是哪一句」。

3. **持久化：Firestore**  
   同一张卡片上的对话会 **保存到云端**（按用户 + 卡片 ID）。下次再打开这张卡片，会先 **拉历史再显示**，所以连 **换设备 / 关掉 App 再开** 也能续上——这仍是 **你们存的数据**，不是模型「记住了用户」。

### 2.3 一句话总结

> **「像记得上一句」= 每次请求都把「完整聊天记录」复制粘贴进当次提示词 + 本地/云端存历史。**  
> 发挥作用的是：**产品设计 + 提示词结构 + 客户端/Firestore 状态**，不是「大模型自带会话记忆」。

## 3. AI 拆解：没有 Agent 为什么也能好用？

AI 拆解（把长文/PDF/链接变成一张张知识卡）主要是：

- **很强的单次指令**：角色、输出格式（标题/正文/闪卡等）、风格（普通/老奶奶/博士/播客等）、字数与结构要求，都写在 **系统或用户提示** 里；
- **输入里带上用户材料**（或解析后的正文）；
- 必要时 **后台任务** 分步（解析 → 生成），但这是 **流水线**，不是 Agent 自主循环。

效果好的原因通常是：**任务边界清晰 + 提示词约束紧 + 模型能力足够**，而不是必须上 AG。

---

## 4. 和 AG（Agent）的对比（方便对外解释）

| | 当前 Reado 做法 | 典型 AG |
|--|------------------|---------|
| 一次用户动作 | 往往 **1 次或固定几步** 模型调用 | 可能 **多轮自主决策**、调搜索/代码/数据库 |
| 记忆 | **显式传入**（正文 + 历史文本） | 常有 **工具 / 记忆组件** |
| 复杂度 | 低，易控成本与稳定性 | 高，要编排与容错 |

所以可以说：**产品是「强提示词 + 状态由 App 管理」的路线，不是 Agent 路线**，同样能把体验做好。

## 5. 相关代码位置（方便开发自查）

- AI 导师：把 `history` 拼进 prompt → `lib/core/services/content_generator_service.dart`（`chatWithContentStream`）
- 消息列表与发请求 → `lib/features/feed/presentation/widgets/feed_item_view.dart`（`_handleSend`、`fetchAiChatHistory` / `saveAiChatHistory`）
- AI 拆解提示与任务 → `lib/data/services/content_extraction_service.dart`、`functions/` 下提示与 Cloud Functions 等

### 5.1 界面语言与 AI 输出语言（一致）

- 用户在 **官网 / 设置** 选择的 **中文或英文** 会写入 **SharedPreferences**（`app_locale_code`），`main` 启动时用 **`LocaleNotifier.fromInitialCode`** 预读，**首帧就是该语言**，避免先英文再跳变。
- **`localeProvider.outputLocale`**（`zh` / `en`）会传给：**AI 拆解/导入**（如 `add_material_modal`、后台 `outputLocale` 字段）、**卡片内 AI 导师**、**整理 Pin 笔记**（`app_prompts.dart` 中英两套系统提示）。
- 登录用户还会把偏好同步到 Firestore **`reado`**：`users/{uid}.preferredLocale` 与分享用的 `share_settings.appLocale`（与 `FirestoreService` 读库一致）。

## 6. 给老板的口头版（30 秒）

「我们没用 Agent 框架。AI 导师看起来能接着聊，是因为 **每次提问都会把整段聊天记录和卡片内容一起发给模型**，对话还存在我们自己服务器上，所以又准又连贯。拆解则是 **一次把规则和材料说清楚**，靠提示词把质量拉起来。」

## 7. 演进路线：从「单卡导师」到「更像真老师」（可选规划）

**目标**：导师不仅记得**当前卡片**里的对话，还能大致知道用户**最近在学什么、掌握到哪**，更像全科老师——**不必一上来就做 Agent**。

### 7.1 是否必须上 Agent？

| 目标 | 是否必须 Agent | 常见做法 |
|------|----------------|----------|
| 知道最近在学哪个模块、进度 | 否 | 读 Firestore 进度/最近访问，拼成一小段 **学习快照** 放进 system 提示 |
| 单卡内接着聊 | 已有 | 对话历史 + 卡片正文 |
| 跨卡片、跨模块串讲 | 有帮助 | 定期生成 **学习画像摘要**，或轻量 **检索最近笔记/卡片标题** |
| 自动规划课表、多工具循环 | 才接近 Agent | 用户量和场景复杂后再上 |

**结论**：「老师感」主要来自 **结构化记忆 + 写进提示词**；Agent 是加分项，不是前提。

### 7.2 建议分阶段（由轻到重）

- **阶段 A（低成本）**  
  登录或定时生成 **学习快照**（最近在学模块名、掌握标签、常问主题关键词）。  
  **AI 导师每次请求多带一小段**，入口仍是卡片内导师，上下文更富。

- **阶段 B**  
  新增 **「全局导师」入口**：不绑单卡，默认带学习快照 + 可选最近笔记摘要；仍是一次/两次模型调用，非 Agent 循环。

- **阶段 C**  
  向量库存卡片/笔记摘要，**先检索再回答**（真 RAG）；再往后才考虑多步 Agent（查进度 → 选题 → 讲解）。

### 7.3 产品话术（对内）

> 先做 **学习画像 + 注入提示词**，验证「老师认识你」的价值；再决定是否上独立入口与检索/Agent。
